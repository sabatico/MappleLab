# Multi-User Implementation Plan

## Current State (what exists)

- `User` model: `username`, `password_hash`, `is_admin` — no email, no quotas
- Open `/auth/register` route — anyone can self-register (must be removed)
- Login form uses "username" label (must become "email")
- Navbar account dropdown has only Logout
- `VM` model: no disk size field
- No email library in `requirements.txt`
- No Flask-Migrate (schema changes done manually so far)

## Phase 2 — Schema Changes

### `[app/models.py](app/models.py)`

**User model additions:**

- `email` — String(255), unique, nullable=True (nullable for existing admin migration safety)
- `max_active_vms` — Integer, default=1
- `max_saved_vms` — Integer, default=2
- `disk_quota_gb` — Integer, default=100
- `must_set_password` — Boolean, default=False (first-login trigger)
- `invite_token` — String(128), nullable=True (UUID, cleared after use)
- `invited_at` — DateTime, nullable=True (token expiry: 72h)
- `last_login_at` — DateTime, nullable=True

**VM model addition:**

- `disk_size_gb` — Float, nullable=True (recorded at save time from `tart list SizeOnDisk`)

**New `AppSettings` model** (single-row config table, keyed by `id=1`):

```python
class AppSettings(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    smtp_host = db.Column(db.String(256), nullable=True)
    smtp_port = db.Column(db.Integer, default=587)
    smtp_user = db.Column(db.String(256), nullable=True)
    smtp_password = db.Column(db.String(256), nullable=True)
    smtp_from = db.Column(db.String(256), nullable=True)
    smtp_use_tls = db.Column(db.Boolean, default=True)
```

Run: `flask db migrate -m "multi-user"` then `flask db upgrade`.

## Phase 3 — Admin Blueprint

Create new blueprint `app/admin/` with routes:

**User management (`/admin/users`):**

- `GET /admin/users` — paginated user list (email, type, quotas, last login, status)
- `POST /admin/users/create` — create user: takes email, role (user/admin), optional quota overrides; generates `invite_token`, saves `must_set_password=True`, sends invite email; `username` set to email prefix
- `GET/POST /admin/users/<id>/edit` — change role, quotas
- `POST /admin/users/<id>/resend-invite` — regenerate token, resend email
- `POST /admin/users/<id>/delete` — delete user and their VMs (with confirmation)

**Settings (`/admin/settings`):**

- `GET/POST /admin/settings` — SMTP config form (reads/writes `AppSettings` row `id=1`)

All routes protected by `@admin_required` decorator (already exists in `app/nodes/routes.py`, move to shared location in `app/utils.py`).

Templates:

- `app/templates/admin/users.html` — user table + Create User inline form
- `app/templates/admin/settings.html` — SMTP form with test-email button

## Phase 6 — Quota Enforcement

### In `[app/main/routes.py](app/main/routes.py)`

`**create_vm()` — check active VM quota:**

```python
active_count = VM.query.filter_by(user_id=current_user.id).filter(
    VM.status.in_(['creating','running','stopped','failed','pushing','pulling'])
).count()
if active_count >= current_user.max_active_vms:
    flash(f'Active VM limit ({current_user.max_active_vms}) reached.', 'danger')
    return ...
```

`**resume_vm()` — same active count check** before pulling.

`**save_vm()` — disk quota check before triggering push:**

1. Query node agent for current VM's disk size (`SizeOnDisk` from `tart list` output, in GB)
2. Sum `disk_size_gb` of user's `archived` VMs from DB
3. If `sum + current_vm_size > user.disk_quota_gb` → flash error, abort
4. If `count(archived VMs) >= user.max_saved_vms` → flash error, abort
5. Store `vm.disk_size_gb = current_vm_size` on the record before commit

## Phase 9 — Admin Operational Dashboard

Add an admin-only operational page to control user VMs without switching accounts.

### Route and navigation

- New page: `GET /admin/overview` (UI label: `Dashboard`)
- All users have `My VMs` tab for their personal VM list
- Admin users additionally have `Dashboard` for cross-user operations

### Dashboard page behavior

For each user, show grouped VM sections for:

- `running`
- `stopped`
- `archived`
- `pushing`
- `pulling`
- `failed`

Each VM row should include:

- VM name
- Current node
- Configured memory (`memory_mb`)
- Configured CPU (`cpu`)
- Saved disk usage (`disk_size_gb`)
- Status-aware controls

### Admin VM controls from overview

- `running`: `Stop`, `Archive`, `Delete`
- `stopped`: `Start`, `Archive`, `Delete`
- `archived`: `Resume`, `Delete`
- `failed`: `Start`, `Re-Pull`, `Delete` (as applicable to row metadata)
- `pushing` / `pulling`: operational visibility plus status-appropriate controls

### Admin control endpoints

- `POST /admin/vms/<vm_id>/start`
- `POST /admin/vms/<vm_id>/stop`
- `POST /admin/vms/<vm_id>/archive`
- `POST /admin/vms/<vm_id>/resume`
- `POST /admin/vms/<vm_id>/repull`
- `POST /admin/vms/<vm_id>/delete`

Implementation notes:

- Archive/Resume flows should reuse existing registry-tag sanitization.
- Archive should keep registry free-space preflight checks.
- Resume should use `find_best_node()` to avoid node congestion.
- Dashboard renders a one-time operation progress snapshot for in-progress rows (no continuous polling on this page).

## Migration Note for Live DB

Since `instance/orchard_ui.db` exists and is excluded from git, the user must run:

```bash
flask db init       # only once if migrations/ folder doesn't exist
flask db migrate -m "multi-user"
flask db upgrade
```

The existing admin user row will gain the new nullable columns with `NULL` values — no data loss. Admin can then set their email via the edit page, or it auto-populates on next login.
