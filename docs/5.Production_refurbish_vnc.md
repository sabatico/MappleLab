# 5. Production Refurbish — VNC

## Goal

Make VNC console access production-ready for corporate LAN/VPN users while keeping node hosts private and ensuring browser compatibility with Apple ARD (`RFB 003.889`) authentication.

## Root Cause Summary

- Browser access works from manager localhost but fails remotely over plain HTTP.
- noVNC ARD auth path requires WebCrypto (`window.crypto.subtle`), which is restricted in non-secure browser contexts.
- Current flow exposes dynamic websocket ports and depends on mixed transport assumptions.

## Target Production Architecture

Use a single secure entrypoint on the manager host:

- Browser -> `https://manager` (UI)
- Browser -> `wss://manager/console/ws/<vm_name>` (same-origin websocket gateway; default)
- Manager -> node `websockify` (direct WS by default on LAN, optional SSH tunnel)
- Node `websockify` -> VM VNC endpoint (`vm_ip:5900`, with configurable preference)

Benefits:

- Secure context guaranteed for noVNC (WebCrypto available).
- Browsers do not need direct reachability to node hosts.
- TLS, auth policy, and observability centralized on manager.

Optional acceleration mode (advanced):
- Browser-direct node websocket (bypass manager WS relay).
- Requires TLS-capable node endpoint (`wss://...`) when manager UI is loaded over HTTPS.

## Current Implementation State

- Same-origin WS gateway is implemented and used by default.
- LAN default runs without SSH tunnel (`VNC_USE_SSH_TUNNEL=false`) to remove tunnel overhead.
- Optional browser-direct mode exists (`VNC_BROWSER_DIRECT_NODE_WS=true`) for environments with node-side TLS websocket endpoints.
- VNC console has live profile switch in UI:
  - Optimize Bandwidth (default)
  - Optimize Render

## Implementation Plan

### Phase 1: Manager Transport Hardening

1. Add a same-origin websocket bridge endpoint in manager app:
   - `GET /console/ws/<vm_name>` upgrade to websocket
   - Proxy frames to the existing local tunnel socket for that VM
2. Update console client URL creation:
   - Use `wss://<manager>/console/ws/<vm_name>` (or `ws://` in local dev only)
3. Stop exposing dynamic local tunnel port numbers to browser templates.

### Phase 2: Deployment Hardening

1. Run Flask app under Gunicorn.
2. Put Nginx or Caddy in front for TLS termination.
3. Enable websocket upgrade routing for `/console/ws/*`.
4. Force HTTPS redirect and secure headers:
   - HSTS
   - `X-Forwarded-Proto`
   - proper websocket proxy headers

### Phase 3: Flask Security Config

Set production-safe values:

- `SESSION_COOKIE_SECURE = True`
- `SESSION_COOKIE_HTTPONLY = True`
- `SESSION_COOKIE_SAMESITE = "Lax"` (or stricter as policy allows)
- `PREFERRED_URL_SCHEME = "https"`

### Phase 4: Node Agent Stability

1. Keep VNC endpoint strategy configurable:
   - `VNC_TARGET_PREFERENCE=vm_ip_first` (default for current Tart behavior)
   - fallback to localhost only if selected or needed
2. Keep probe and selected-target logs for diagnostics.
3. Retain actionable errors for no reachable endpoint.

## Verification Plan

1. Open UI from a remote user machine over `https://manager`.
2. Open VM console and confirm desktop renders (not grey).
3. Confirm browser console has no `crypto.subtle`/secure-context errors.
4. Confirm manager logs websocket bridge established and clean close.
5. Confirm node logs show expected VNC target selection and websockify startup.

## Rollout Notes

- Maintain a dev mode path that allows local plain HTTP for rapid testing.
- Gate production behavior with env flags (`FLASK_ENV=production` and proxy settings).
- Document reverse proxy snippets and certificate provisioning in README.
